# Lessons Learned

| Date | What went wrong | Rule | Fix |
|------|----------------|------|-----|
| 2026-04-17 | Removed `.label` element between breadcrumb and h1 when mirroring Insights page layout to Careers, but didn't account for the visual gap the label was contributing (~34px). Declared spacing "the same" based on CSS values alone without measuring computed layout in the browser. | When removing or adding elements between siblings, always measure the actual rendered gap (via JS `getBoundingClientRect()`) on both pages before and after. Never assume CSS values tell the full story — hidden elements, margins, padding, and margin collapsing all affect final layout. | Used `getBoundingClientRect()` to measure exact gap on Insights (57.59px), then adjusted careers h1 `margin-top` to `3.6rem` to match exactly. |
| 2026-04-17 | ClickUp dropdown custom fields (Specialist Field, Location) silently rejected submissions because option UUIDs were truncated to the first 8 characters (e.g. `7510c0cb` instead of the full `7510c0cb-1671-4d26-8b01-737e3750ab26`). ClickUp returned error `FIELD_011: Value must be an option index or uuid` but the Worker returned `success: true` to the user anyway. Manual curl tests didn't catch it because they omitted dropdown fields. | Always use full UUIDs from ClickUp — never truncate. When integrating with an external API, verify by testing with **all** fields populated, not just required ones. Log and surface external API errors to the caller instead of swallowing them. | Replaced all 18 partial UUIDs with full UUIDs in `SPECIALIST_OPTIONS` and `LOCATION_OPTIONS`. Added null check on `taskId` after `createClickUpTask` so the Worker returns 502 on failure instead of false success. |
| 2026-04-17 | `createClickUpTask` returned `null` on ClickUp API failure, but both `handleApply` and `handleEOI` didn't check for null before returning `success: true`. Users saw "Thanks for applying!" while their application was silently lost. | Every external API call on the critical path must check its return value. If the operation failed, return an error to the user — never assume success. Best-effort operations (attachment, linking) can fail silently, but the primary operation (task creation) must not. | Added `if (!taskId)` guard returning `{ success: false, message: '...', status: 502 }` in both handlers before proceeding to best-effort operations. |
